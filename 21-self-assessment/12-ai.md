### 1. What are embeddings, and how is cosine similarity used to compare them?

<details>
<summary>Reveal answer</summary>

**Embeddings** are dense vector representations of text (or images, audio) in a high-dimensional space where **semantic similarity is captured by geometric proximity**. A sentence like "The cat sat on the mat" might become a vector of 1536 floats (for OpenAI's `text-embedding-3-small`).

**Cosine similarity** measures the angle between two vectors, ignoring magnitude:

```
cosine_similarity(A, B) = (A . B) / (|A| * |B|)
```

- **1.0** = identical direction (semantically identical).
- **0.0** = orthogonal (unrelated).
- **-1.0** = opposite (rare with modern embeddings).

**Why cosine over Euclidean distance?** Cosine is invariant to vector length -- two documents about the same topic will be "close" even if one is much longer. This makes it the standard metric for semantic search.

</details>

### 2. What is RAG, and what problem does it solve?

<details>
<summary>Reveal answer</summary>

**RAG (Retrieval-Augmented Generation)** solves the problem of LLMs lacking access to **private, current, or domain-specific knowledge**. Instead of fine-tuning (expensive, slow), RAG retrieves relevant documents at query time and injects them into the prompt.

**Flow**:
1. **Ingest**: split documents into chunks, generate embeddings, store in a vector database.
2. **Query**: embed the user's question, search the vector DB for top-K similar chunks.
3. **Generate**: pass the retrieved chunks as context in the prompt, and the LLM generates an answer grounded in that context.

**Chunking strategies**:
- **Fixed-size** (e.g., 512 tokens with overlap) -- simple but may split mid-sentence.
- **Semantic** -- split on paragraph/section boundaries to preserve meaning.
- **Recursive** -- try larger separators first (sections), fall back to smaller (sentences).

**Hybrid search** combines vector search (semantic similarity) with keyword search (BM25/full-text) for better recall -- semantic search catches paraphrases while keyword search catches exact terms.

Deep dive: [RAG](../19-ai/05-rag.md)

</details>

### 3. What is a system prompt, and how do few-shot examples and Chain-of-Thought improve output quality?

<details>
<summary>Reveal answer</summary>

- **System prompt** -- sets the LLM's persona, constraints, and output format. It runs before the user's message and shapes all subsequent responses. Example: "You are a senior .NET developer. Respond only in C# with XML comments."

- **Few-shot examples** -- include 2-5 input/output examples in the prompt so the model learns the pattern by demonstration, without fine-tuning:

```
User: Translate "hello" to French.
Assistant: bonjour

User: Translate "goodbye" to French.
Assistant: au revoir

User: Translate "thank you" to French.
```

- **Chain-of-Thought (CoT)** -- instruct the model to "think step by step". This dramatically improves accuracy on reasoning tasks (math, logic, multi-step analysis) by forcing the model to show intermediate steps rather than jumping to an answer.

**Hallucinations** are when the model generates plausible-sounding but factually incorrect information. Mitigation strategies: RAG (ground in real data), lower temperature, ask the model to cite sources, and add "If you don't know, say so" to the system prompt.

Deep dive: [Prompt Engineering](../19-ai/03-prompt-engineering.md)

</details>

### 4. How does function calling work in LLMs, and what are the security concerns?

<details>
<summary>Reveal answer</summary>

**Flow**:
1. You define available functions (name, description, JSON schema for parameters) and send them with the user's message.
2. The LLM decides whether to call a function and generates a JSON object with the function name and arguments.
3. **Your code** executes the function (API call, DB query, etc.) and returns the result.
4. The LLM incorporates the result into its final response.

The LLM **never executes code** -- it only produces structured JSON describing the call. Your application is the executor.

**Security concerns**:
- **Prompt injection** -- a malicious user could trick the model into calling dangerous functions (e.g., "delete all records"). Always validate and authorize on the server side.
- **Over-permissioned functions** -- don't expose destructive operations without confirmation workflows.
- **Data exfiltration** -- a function that sends emails could be abused to leak context window data. Use allowlists and rate limiting.
- **Input validation** -- never trust the LLM's generated arguments blindly; validate types, ranges, and permissions before executing.

Deep dive: [OpenAI API](../19-ai/02-openai-api.md)

</details>

### 5. What is Semantic Kernel, and what are its core building blocks?

<details>
<summary>Reveal answer</summary>

**Semantic Kernel (SK)** is Microsoft's open-source SDK for integrating LLMs into .NET (and Python/Java) applications. It provides an orchestration layer between your code and AI models.

**Core building blocks**:

- **Kernel** -- the central object that holds services (LLM connectors), plugins, and configuration. It's the entry point for all AI operations.
- **Plugins** -- collections of functions the AI can call. Two types:
  - **Prompt functions** -- templated prompts (text + parameters).
  - **Native functions** -- C# methods decorated with `[KernelFunction]`.
- **Auto function calling** -- the Kernel can automatically detect when the LLM wants to call a function, execute it, and feed the result back. Set `ToolCallBehavior = ToolCallBehavior.AutoInvokeKernelFunctions`.
- **Filters** -- intercept function calls before/after execution for logging, validation, authorization, or modification. Types: `IFunctionInvocationFilter`, `IPromptRenderFilter`, `IAutoFunctionInvocationFilter`.
- **Planners** (deprecated in favor of auto function calling) -- previously used to create multi-step plans.

```csharp
var kernel = Kernel.CreateBuilder()
    .AddAzureOpenAIChatCompletion(deployment, endpoint, apiKey)
    .Build();
kernel.Plugins.AddFromType<TimePlugin>();
```

Deep dive: [Semantic Kernel](../19-ai/04-semantic-kernel.md)

</details>

### 6. What is MCP (Model Context Protocol), and how does it differ from function calling?

<details>
<summary>Reveal answer</summary>

**MCP** is an open protocol (by Anthropic) that standardizes how AI applications connect to external data sources and tools. Think of it as a **USB-C for AI** -- a universal interface.

**Architecture**:
- **Host** -- the AI application (e.g., Claude Desktop, an IDE).
- **Client** -- manages a 1:1 connection with an MCP server.
- **Server** -- exposes tools, resources, and prompts over a standardized transport (stdio or HTTP+SSE).

**MCP vs Function Calling**:

| Aspect | Function Calling | MCP |
|--------|-----------------|-----|
| Definition | Defined per API call in the prompt | Defined by the server, discovered dynamically |
| Scope | Single LLM interaction | Cross-application, cross-model |
| Execution | Your app executes | MCP server executes |
| Discovery | Static (you list functions) | Dynamic (client queries server capabilities) |
| Standard | Provider-specific (OpenAI, Anthropic) | Open protocol, vendor-agnostic |

MCP is higher-level: it lets you build **reusable tool servers** that any MCP-compatible host can discover and use, rather than wiring functions into each application.

Deep dive: [MCP](../19-ai/06-mcp.md)

</details>

### 7. How do you manage token limits and costs when building LLM-powered applications?

<details>
<summary>Reveal answer</summary>

**Token management strategies**:

1. **Context window budgeting** -- allocate portions: system prompt (fixed), conversation history (sliding window), RAG context (dynamic), response (reserved).
2. **Conversation pruning** -- summarize older messages instead of keeping full history. Keep the system prompt and last N turns.
3. **Chunking for RAG** -- retrieve only the most relevant chunks, not entire documents. Use `top_k` wisely (3-5 chunks is often enough).
4. **Model selection** -- use cheaper/smaller models for simple tasks (classification, extraction) and expensive models for complex reasoning.
5. **Caching** -- cache embeddings and common query results. Semantic caching can detect similar (not identical) questions.

**Cost control**:
- Set `max_tokens` to limit response length.
- Use streaming to detect and abort runaway responses.
- Implement per-user rate limits and token budgets.
- Monitor usage with dashboards (tokens consumed per endpoint, per user).
- Prompt optimization -- shorter, more precise prompts reduce input tokens.

Deep dive: [OpenAI API](../19-ai/02-openai-api.md)

</details>

### 8. Why does streaming matter for LLM applications?

<details>
<summary>Reveal answer</summary>

LLMs generate tokens sequentially, so a full response might take 5-30 seconds. Without streaming, the user sees **nothing** until the entire response is generated.

**With streaming**:
- Tokens arrive as they're generated (Server-Sent Events or WebSocket).
- The user sees the response forming in real-time -- **perceived latency drops from seconds to milliseconds**.
- Time to First Token (TTFT) becomes the key metric, not total generation time.

**Implementation pattern in .NET**:

```csharp
await foreach (var chunk in client.GetStreamingChatCompletionsAsync(options))
{
    var content = chunk.ContentUpdate;
    await response.WriteAsync(content);
    await response.Body.FlushAsync();
}
```

**Additional benefits**:
- You can **abort early** if the output goes off-track (saves tokens and money).
- Better UX for long responses (documentation, code generation).
- Works naturally with SignalR for real-time web applications.

Deep dive: [OpenAI API](../19-ai/02-openai-api.md)

</details>

### 9. What chunking strategies exist for RAG, and how do you choose?

<details>
<summary>Reveal answer</summary>

| Strategy | How it works | Pros | Cons |
|----------|-------------|------|------|
| **Fixed-size** | Split every N tokens with overlap | Simple, predictable | Splits mid-sentence/concept |
| **Recursive character** | Try splitting by `\n\n`, then `\n`, then `. `, then space | Respects document structure | May produce uneven chunks |
| **Semantic** | Use embeddings to detect topic boundaries | Best coherence | Slower, more complex |
| **Document-aware** | Use headings, page breaks, code blocks | Preserves logical sections | Requires format-specific parsers |

**Choosing**:
- Start with **recursive character splitting** (LangChain/Semantic Kernel default) -- it works well for most text.
- Use **document-aware** for structured content (Markdown, HTML, code).
- Add **overlap** (10-20%) to avoid losing context at chunk boundaries.
- **Chunk size** depends on your model's context window and retrieval precision. Smaller chunks (256-512 tokens) give better precision; larger chunks (1024+) give better context.

**Test empirically**: the best strategy depends on your specific documents and queries.

Deep dive: [RAG](../19-ai/05-rag.md)

</details>

### 10. How would you test an AI-powered feature? What strategies apply?

<details>
<summary>Reveal answer</summary>

AI outputs are **non-deterministic**, so traditional assertion-based testing needs adaptation:

1. **Set temperature to 0** for reproducible tests (not perfectly deterministic, but close).
2. **Evaluation datasets** -- create input/expected-output pairs and measure quality metrics (accuracy, F1, BLEU, human rating).
3. **LLM-as-judge** -- use a separate LLM to evaluate the quality of another LLM's output against criteria (relevance, accuracy, tone).
4. **Guardrail tests** -- verify the system rejects prompt injection, stays on topic, respects output format constraints.
5. **Integration tests for tool calling** -- verify the correct function is called with valid arguments for known inputs.
6. **Semantic similarity tests** -- embed the expected and actual outputs and assert cosine similarity is above a threshold.
7. **Regression suites** -- save golden examples and detect when model updates or prompt changes degrade quality.
8. **Cost and latency tests** -- assert token usage and response time stay within budgets.

**What NOT to test**: exact string matching on LLM output (too brittle). Focus on **structure, intent, and constraints** rather than exact wording.

Deep dive: [AI Architecture](../19-ai/07-ai-architecture.md)

</details>
